ACT I · THE PARADIGM

I.

RAG

Not a vector database. A pattern for grounding generation in retrieval.

ACT I · CONCEPT 1 OF 12
WHAT IS RAG

Retrieval-Augmented Generation is a retrieval pattern,
not a vector database.

RAG is the architectural discipline of fetching relevant context from external sources before asking an LLM to generate a response. The source of that context must be dictated by the nature of the question — never by the limitations of the tool.

The Misconception

2023 · THE TOY
  • One PDF, chunked, embedded, retrieved
  • Vector DB treated as the answer to everything
  • Numeric questions answered with semantic similarity
  • Relationship questions answered by keyword match
  • Hallucinations attributed to “the model”

The Pattern

2026 · THE DISCIPLINE
  • Multiple sources, each fit for its question class
  • Text-to-SQL for facts, vectors for context, graphs for links
  • A router decides where the question goes
  • Pydantic contracts enforce the answer shape
  • Hallucinations attributed to poor retrieval
THE THESIS · EVERY ACT THAT FOLLOWS
An enterprise RAG is judged by where it looks, not what it embeds. The rest of this deck is the engineering that makes that real.
ACT I · CONCEPT 2 OF 12
RAG EVOLUTION · FOUR GENERATIONS

The pattern stopped being a pipeline.
It became a loop with judgment.

Each generation kept the previous one’s primitives and added a layer of intelligence around them. By 2026 the system decides for itself whether the retrieval was good enough.

GEN 1 · 2022–23

NAIVE

Chunk → embed → cosine search → stuff into prompt. One index, one shot, no judgment.

single vector DB · static k
GEN 2 · 2023–24

ADVANCED

Pre/post-retrieval tuning, re-ranking, query expansion, hybrid keyword + vector search.

same pipeline · better tuning
GEN 3 · 2024–25

MODULAR

Interchangeable modules, multiple indexes, query routing — pick the right index per question.

SQL + Vector + Graph · routed
GEN 4 · 2026 · THIS WORKSHOP
Agentic. The retrieval itself becomes a loop.

Agents decide when to retrieve, where to retrieve, judge the result, and iterate. No more single-shot. The system answers when it’s confident.

ACT I · CONCEPT 5 OF 12 · STEP 1 / 9
OFFLINE · LOAD

Step 1 · Load. Readers turn sources into Document objects.

Every source — Postgres rows, Mongo collections, S3 PDFs — is normalized into the same atomic unit: a Document with raw text plus metadata. Nothing downstream cares where it came from.

SOURCE 1 PostgreSQL customers · orders · products
·
SOURCE 2 MongoDB event_logs · user_activity
·
SOURCE 3 SeaweedFS PDFs · CSVs · manuals
↓ ↓ ↓
READERS · ONE PER SOURCE Connectors DatabaseReader · SimpleMongoReader · S3Reader pluggable, swap any reader without touching downstream
→
OUTPUT Document .text · .metadata · .doc_id TO STEP 2
THE INVARIANT
After Step 1, everything downstream sees a uniform stream of Document objects. The choice of database vanishes. The pipeline is now polymorphic.
ACT I · CONCEPT 6 OF 12 · STEP 2 / 9
OFFLINE · CHUNKING / SPLITTING

Step 2 · Split. The unit of retrieval is the chunk, not the document.

Embeddings have a token ceiling and retrievers return passages, not files. A bad splitter cuts a sentence in half — the embedding loses meaning, the retrieval loses precision.

FROM STEP 1 Document raw text + metadata
→
SPLITTER Cut Boundaries SentenceSplitter or SemanticSplitterNodeParser fixed-size or embedding-aware
→
OUTPUT Node[ ] list of chunks — the unit of retrieval TO STEP 3

SentenceSplitter

FIXED-SIZE · THE BASELINE
  • Configurable chunk_size and chunk_overlap
  • Respects sentence boundaries when it can
  • Predictable cost, predictable token count
  • Cuts mid-paragraph when topics shift
  • Use when: tight latency budget, uniform documents

SemanticSplitterNodeParser

EMBEDDING-AWARE · THE DEFAULT
  • Embeds sentences, cuts when embedding distance spikes
  • Chunk boundaries follow meaning, not character count
  • Variable chunk size — richer retrieval
  • Costs extra embeddings at ingestion time
  • Use when: mixed-topic docs, retrieval quality matters
CLAUDE.md RULE · THIS PROJECT
Semantic chunking is the default. SentenceSplitter is only used when a measured benchmark proves it wins on a specific source.
ACT I · CONCEPT 7 OF 12 · STEP 3 / 9
OFFLINE · METADATA ENRICHMENT

Step 3 · Enrich. Let an LLM read the chunk so the retriever doesn’t have to.

Metadata extractors pre-compute what a chunk is about — not just what it contains. Retrieval at query time becomes intent-matching, not keyword-matching.

FROM STEP 2 Node[ ] raw chunks
→
EXTRACTORS LLM Pass Title · Summary · QuestionsAnswered one LLM call per chunk per extractor
→
OUTPUT Enriched Node chunk + title + summary + questions TO STEP 4

TitleExtractor

What is this chunk's title?
LLM proposes a short title for each chunk. Becomes a metadata field used at retrieval to bias toward on-topic chunks.

SummaryExtractor

What does it say in one sentence?
A compact summary attached to the chunk. Lets the retriever match a query against a distilled abstract rather than raw prose.

QuestionsAnsweredExtractor

What can it answer?
LLM generates 3–5 questions the chunk could answer. The query is matched against those — intent-first retrieval.
THE TRADE-OFF
Enrichment is expensive at ingestion time (one LLM call per chunk per extractor) and free at query time. For high-traffic systems, it pays for itself in days.
ACT I · CONCEPT 8 OF 12 · STEP 4 / 9
OFFLINE · EMBED

Step 4 · Embed. Text becomes geometry.

Each chunk is passed through an embedding model and returned as a fixed-length vector. Similar meanings live near each other in that high-dimensional space — that’s what makes retrieval possible.

FROM STEP 3 Enriched Node chunk + metadata
→
EMBED MODEL Encoder text-embedding-3-large · bge-large text in — fixed-length vector out
→
OUTPUT Vector 1536-dim point in semantic space TO STEP 5
# conceptually chunk = "The billing pipeline runs every 6 hours and writes to the invoices table…" vector = embed_model.encode(chunk) # → [ 0.024, -0.118, 0.082, 0.331, -0.057, …, 0.014 ] # 1536 floats
THE INSIGHT
Similarity search is just geometry. Two chunks are “similar” if their vectors are close. The quality of your retrieval is bounded by the quality of your embedding model.
ACT I · CONCEPT 9 OF 12 · STEP 5 / 9
OFFLINE · INDEX & STORE

Step 5 · Index. One index per cognitive domain.

The chunk + vector + metadata triple gets persisted into the store that matches its domain. LlamaIndex gives one abstraction per store — same pipeline, three destinations.

FROM STEP 4 Chunk + Vector enriched node
→
LLAMAINDEX Index Abstraction SQLDatabase · VectorStoreIndex · PropertyGraphIndex one index per cognitive domain
→
PERSISTED 3 Stores Postgres · Qdrant · Neo4j READY TO QUERY

THE PAYOFF
Five steps in, the system is ready to answer. The next four steps are runtime — they run on every user question.
ACT I · CONCEPT 10 OF 12 · STEP 6 / 9
RUNTIME · QUERY

Step 6 · Query. The question enters the system.

A typed request crosses the boundary. From here on, latency is the budget — everything must happen in the time the user is willing to wait.

USER Natural Lang. “Which enterprise customers…?”
→
BOUNDARY QueryRequest Pydantic BaseModel question + optional source override
→
ENGINE RAG Entry SubQuestionQueryEngine awaits routing decision TO STEP 7
class QueryRequest(BaseModel): question: str sources: list[str] | None = None # optional override
THE CONTRACT BOUNDARY
The request is already a Pydantic model. We refuse dict and Any at the API boundary — the contract is the API.
ACT I · CONCEPT 11 OF 12 · STEP 7 / 9
RUNTIME · ROUTE & RETRIEVE

Step 7 · Route. An LLM picks the engine — or splits the question.

This is the moment where multi-source RAG earns its name. A small LLM call decides which engine — or, for cross-domain questions, decomposes the question into sub-questions and routes each one.

FROM STEP 6 Question single user query
→
DECOMPOSE Sub-Questions LLM splits cross-domain query n sub-questions, each typed by domain
→
ROUTE 3 Engines Ledger · Memory · Brain parallel retrieval, one per domain
→
FUSE Joined Context SQL rows + passages + graph paths TO STEP 8

RouterQueryEngine

SINGLE-INTENT QUESTIONS
  • One LLM call: “which engine fits this question?”
  • Routes the full question to one downstream engine
  • Cheap — one extra LLM call before retrieval
  • Used for: “How many enterprise customers do we have?”
  • Verdict: Ledger. SQL. Done.

SubQuestionQueryEngine

CROSS-DOMAIN QUESTIONS · THE STAR
  • Decomposes the question into n sub-questions
  • Each sub-question is routed independently
  • Engines run in parallel, results are joined
  • Used for: “Which enterprise customers had a billing incident, and what tables would break if rolled back?”
  • Verdict: Ledger + Memory + Brain — all three.
THE ARCHITECTURAL RULE
SubQuestionQueryEngine is the default for this workshop. The cross-domain question is the one worth optimizing for.
ACT I · CONCEPT 12 OF 12 · STEP 8 / 9
RUNTIME · SYNTHESIZE

Step 8 · Synthesize. Fragments become a coherent answer.

Retrieval returns scattered passages, rows, and graph paths. Synthesis is the LLM’s job: weave them into one prose answer that cites its sources.

FROM STEP 7 Fragments rows + passages + paths
→
STRATEGY Synthesizer tree_summarize or refine parallel-recursive or sequential-refining
→
OUTPUT Prose Answer coherent text + source citations TO STEP 9

tree_summarize

PARALLEL · HIGH-RECALL
  • Summarizes each retrieved chunk independently
  • Then summarizes the summaries — recursively
  • Parallelizable, fast on many chunks
  • Use when: many sources, want breadth
  • Risk: shallow on any one source

refine

SEQUENTIAL · HIGH-PRECISION
  • Draft an answer from the first chunk
  • Refine it incrementally as more chunks arrive
  • Slower — chunks must process in series
  • Use when: precision matters more than latency
  • Risk: drift if the order is wrong
A NOTE ON SOURCES
Synthesis must preserve provenance. Every claim in the answer carries the chunk(s) it came from — the user (and the validator in Step 9) can trace any sentence back to its source.
ACT I · STEP 9 / 9
RUNTIME · VALIDATE

Step 9 · Validate. Pydantic forces the LLM to color inside the lines.

The LLM returns prose; downstream consumers need structure. Pass a Pydantic schema as output_cls= — LlamaIndex retries until the response parses or raises.

FROM STEP 8 Prose Answer LLM-generated text
→
PARSE Pydantic output_cls=QueryResponse retry on parse failure · raise on exhaustion
→
VALIDATED QueryResponse answer + sources + confidence TO API CONSUMER
from pydantic import BaseModel, Field from llama_index.core.program import LLMTextCompletionProgram class QueryResponse(BaseModel): answer: str = Field(..., description="synthesized answer") sources: list[str] = Field(default_factory=list) sub_questions: list[str] | None = None confidence: float = Field(ge=0.0, le=1.0) program = LLMTextCompletionProgram.from_defaults( output_cls=QueryResponse, llm=llm, prompt_template_str=PROMPT, ) result: QueryResponse = program(query_str=user_question)

Why this is non-negotiable

CLAUDE.md · ARCHITECTURE RULE
  • Downstream consumers (MCP, FastAPI, agents) are typed
  • A loose dict breaks them silently
  • Pydantic catches malformed LLM output at the boundary
  • Retries automatic; failures explicit
  • The contract is the API

PROTOTYPING & DEV ENVIRONMENTS

Right tool, right phase. Wrong tool, wrong scale.

LangFlow is a phase-1 tool: discovery, demos, stakeholder alignment. When the pipeline earns the right to become product, you graduate to code — LlamaIndex, Pydantic, tests.

REACH FOR LANGFLOW WHEN…
Discovery & Demos
  • You are sketching a RAG to see if the idea is worth building
  • Stakeholders want to see the data flow, not read code
  • You need to A/B prompts or embedders on a small corpus, fast
  • You are teaching — the graph is the curriculum
  • Lifecycle: hours to days, then archived or graduated
REACH FOR CODE WHEN…
Production & Scale
  • The system must pass tests in CI
  • You need typed contracts (Pydantic) at every boundary

  • Lifecycle: months to years, owned by a team

ACT IV · CONCEPT 3 OF 6
THE FRAMEWORK · FIVE ABSTRACTIONS, ONE FLOW

Five abstractions you can name. Two phases you can split.

Top band is offline (built once, replayed when sources change). Bottom band is runtime (executed on every question). Each abstraction in our codebase is named below it.

ACT IV · MECHANIC 1 OF 5
SETTINGS · THE GLOBAL WIRE

Wire it once. Every component reads from the same hub.

Settings is a process-global singleton. Set it at boot — every Reader, Pipeline, Index, Engine, and Synthesizer reads from it automatically. No more passing llm= through six layers.

Read the rings outward

EACH LAYER INHERITS THE CORE

Set the core fields once; every ingestion component picks up the embed model and node parser; every runtime engine picks up the LLM. Override only when you need a different dependency.

Replaces ServiceContext

THE MIGRATION

Old API: ServiceContext.from_defaults(…) + set_global_service_context(). Deprecated. Same fields, same semantics — just attribute assignment on Settings now.

Every query engine is three stages, in order.
Retrieve · Postprocess · Synthesize.

The stock RetrieverQueryEngine is composable from three swappable parts. The internal method below is literally two lines — everything interesting happens in the components you plug in.

STAGE 1
Retriever
VectorIndexRetriever(similarity_top_k=10)

ACT IV · MECHANIC 3 OF 5
RESPONSE SYNTHESIZERS · SIX MODES, ONE FACTORY

Six ways to fold retrieved chunks into one answer.

One factory: get_response_synthesizer(response_mode=…). Six strategies, each making a different latency vs precision vs coverage trade-off.

COMPACT
Repack chunks into the largest prompts that fit, then refine across them.
fewest calls cheap
Use when: generic Q&A — the default that wins most days.
REFINE
Draft from chunk 1, then refine sequentially as more chunks arrive.
precise sequential
Use when: precision > latency — e.g. compliance answers.
TREE_SUMMARIZE
Summarize each chunk in parallel, then summarize the summaries — recursively.
parallel long answers
Use when: many chunks, want a long synthesized narrative.
SIMPLE_SUMMARIZE
Truncate all chunks into one prompt and answer in a single call.
one call data loss
Use when: latency-critical, short context, willing to truncate.
ACCUMULATE
Apply the prompt to each chunk separately and concatenate the results.
N calls per-source answer
Use when: you want one answer per chunk — e.g. multi-doc Q&A.
GENERATION
Ignore the retrieved nodes; answer the query from the LLM’s prior knowledge.
no RAG debug only
Use when: ablation / baseline / debugging retrieval quality.
ACT IV · MECHANIC 4 OF 5
WORKFLOWS · THE @STEP DECORATOR

Pipelines stop being hard-coded.
They become typed event graphs.

A Workflow is a class of @step-decorated async methods. Each step consumes a typed event and emits another — the framework wires the graph from the type signatures. State flows through ctx.store.

from llama_index.core.workflow import ( Workflow, Context, step, StartEvent, StopEvent, Event, ) class Classified(Event): engines: list[str] class Synthesize(Event): results: list[str] class RouterWorkflow(Workflow): @step async def classify(self, ctx: Context, ev: StartEvent) -> Classified: await ctx.store.set("q", ev.query) return Classified(engines=["ledger", "memory"]) @step async def fan_out(self, ctx: Context, ev: Classified) -> Synthesize: q = await ctx.store.get("q") results = [await run(name, q) for name in ev.engines] return Synthesize(results=results) @step async def finish(self, ctx: Context, ev: Synthesize) -> StopEvent: return StopEvent(result=join(ev.results)) answer = await RouterWorkflow().run(query="who and what?")
