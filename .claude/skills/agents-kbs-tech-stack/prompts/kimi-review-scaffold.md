# Cold-Eyes Scaffold Review

## Context

You are reviewing files just generated by `agents-kbs-tech-stack` v0.3.0. Each generated agent must follow specific role conventions. The scaffold renders templated agent files (`<tech>-architect.md`, `<tech>-developer.md`, optionally `<tech>-troubleshooter.md`) and a KB tree (`quick-reference.md`, `concepts/`, `patterns/`, `reference/`) under `<TARGET_REPO>/.claude/`.

Your job is a cold-eyes pass: assume nothing about the scaffold's correctness. Read each generated file, compare it against the rubric below, and surface every violation. Local checks already cover placeholder leaks and role tool-line tokens at a coarse grain ‚Äî your value is catching the **semantic** drifts (e.g., a developer agent whose body talks like an architect, a KB index that references a domain whose directory does not exist, frontmatter that is syntactically valid YAML but missing a required key).

---

## Rubric

### 1. Role boundaries

The role split is the load-bearing convention of this skill. Tools lists must match the role:

- **`*-architect.md`** ‚Äî `tools:` line MUST NOT contain `Bash`. Architects plan; they never execute. They may have `Read, Write, Edit, Grep, Glob, TodoWrite` plus MCPs.
- **`*-developer.md`** ‚Äî `tools:` line MUST contain `Bash`. Developers ship code, run tests, and apply fixes. They get the full toolset.
- **`*-troubleshooter.md`** ‚Äî `tools:` line MUST contain `Bash` but MUST NOT contain `Edit` and MUST NOT contain `Write`. Troubleshooters diagnose read-only; they hand off to the developer for patches.

Beyond the tools line, the **body** should also reflect role. Flag any architect that writes production code blocks as authoritative output, or any developer that fills its body with trade-off matrices and ADR-style decision text instead of patterns + code.

### 2. Frontmatter validity

Every generated agent must have valid YAML frontmatter with these keys present:

- `name` ‚Äî non-empty string, matches the filename slug (e.g., `apache-spark-architect`).
- `description` ‚Äî non-empty string or block scalar.
- `tools` ‚Äî non-empty list (see role boundaries above).
- `model` ‚Äî non-empty string (e.g., `opus`, `sonnet`).
- `color` ‚Äî non-empty string (e.g., `blue`, `green`, `red`).

If frontmatter parses but a key is missing, that is still a BLOCKER ‚Äî Claude Code's agent loader treats missing keys as silent disable.

### 3. Placeholder leak check

The renderer substitutes `{ALL_CAPS_TOKEN}` placeholders from environment variables. Any unrendered `{ALL_CAPS_TOKEN}` left in a generated file is a render bug. Flag every leak with file path and line number.

Note: the local script already greps for this; your job is to confirm and surface any leaks the local pass missed (e.g., inside frontmatter block scalars where regex anchoring can be tricky).

### 4. KB integrity

For each domain registered in `<TARGET_REPO>/.claude/kb/_index.yaml`:

- The `path:` field must point to a directory that actually exists under `kb/`.
- That directory must contain a `quick-reference.md` file.
- Every entry under `concepts:`, `patterns:`, and `reference:` must point to an existing `.md` file.
- The domain's `name:` and `description:` must not be empty or placeholder strings (`<TODO: ...>` is fine in KB entry bodies but not in the index header).

A `_index.yaml` that references a domain whose directory was never created is a BLOCKER ‚Äî closers will fail to ground.

---

## Severity definitions

- **BLOCKER** ‚Äî any rubric violation. The scaffold is not safe to use until fixed. Examples:
  - Architect with `Bash` in tools
  - Developer without `Bash` in tools
  - Troubleshooter with `Edit` or `Write` in tools
  - Missing frontmatter key
  - Unrendered `{PLACEHOLDER}` in a generated file
  - `_index.yaml` references a domain dir that does not exist

- **IMPORTANT** ‚Äî drift or inconsistency that degrades quality but does not break the scaffold. Examples:
  - Architect body contains a long production code block presented as the canonical answer
  - Developer body contains an ADR-style trade-off matrix instead of patterns
  - `name:` in frontmatter does not match the filename slug
  - KB entry has no `<!-- TODO -->` blocks AND no real content (looks abandoned)
  - Memorable maxim is empty or boilerplate

- **NIT** ‚Äî style. Examples:
  - Inconsistent capitalization in headings
  - Trailing whitespace
  - Missing blank line before a fenced code block

## Output format

Return **only** a single JSON object on stdout, no prose around it:

```json
{
  "summary": "One-line synthesis of what you found.",
  "findings": [
    {
      "severity": "BLOCKER | IMPORTANT | NIT",
      "file": "<absolute or repo-relative path>",
      "line": <integer or null>,
      "message": "What is wrong.",
      "suggestion": "How to fix it."
    }
  ]
}
```

If there are no findings, return `{"summary": "Clean scaffold.", "findings": []}`.

Do not wrap the JSON in a code fence. Do not include any explanation outside the JSON. The orchestration script parses your stdout as JSON.
